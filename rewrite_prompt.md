## Требования к контенту

### Транспорт — разумная и полезная информация
Для каждого места/маршрута указывать:
- **Ближайший транспортный узел** (аэропорт или главный город)
- **Логичные способы добраться** от этого узла до места назначения
- **Конкретные детали:**
  - Цены в лирах и/или евро
  - Расписание/частоту движения
  - Номера маршрутов (долмуш/автобус)
  - Где садиться/выходить
  - Время в пути

**Чего не писать:**
Международные перелёты (Москва → Анталья) — это общая информация в статье "Авиасообщение"
Нереалистичные маршруты (например, поезд из России в Турцию)
Все возможные варианты — только разумные

### История — интродакшен
- Не академический курс, но достаточно чтобы понять значение места
- 1 абзац до пол страницы
- Содержательная, интересная

### Конкретика вместо общих фраз
**Плохо:**
- "Школы для всех уровней"
- "Множество ресторанов"
- "Красивые виды"

**Хорошо:**
- "Здесь 3 школы со стабильным боковым ветром 15-25 узлов в сезон"
- "Локанта Mehmet в центре — вкусная и недорогая еда"
- "Панорама на Голубую лагуну открывается с маяка"

### Живой повествовательный стиль
**Плохо:**
- Сухие точечные списки без контекста
- Маркированные перечни "Город — функция"
- Энциклопедический конспект без эмоций

**Хорошо:**
- Абзацы с живыми историями и деталями
- Описания, которые вызывают визуальные образы
- Эмоциональная связь читателя с местом/темой
- Исключение: списки допустимы только там, где это принципиально (например, чек-листы, перечисление коротких однотипных пунктов)

### Линки
- Сохранять существующие внутренние ссылки
- Добавлять новые по мере необходимости
- Проверять корректность при возможности
- Формат: `[Текст ссылки](/путь-к-статье)`

### Длина текста
- Тексты должны быть исчерпывающими для туриста на неделю
- Если старая статья — 100 строк, новая — 300-500 строк
- Но без воды — только полезная информация

## Как пользоваться системой отслеживания прогресса

### Файл progress.md

Я создам файл `progress.md` в директории проекта, который содержит список всех 82 статей с чекбоксами. Это файл позволяет:
- Отслеживать, что уже переписано
- Возобновлять работу после прерываний
- Визуализировать общий прогресс

### Легенда статусов

- **[ ]** — статья не начата
- **[⏳]** — статья готова, лежит в папке content/articles/new/ — на проверке у пользователя
- **[✓]** — статья одобрена пользователем

### Workflow (Как мы будем работать)

```
1. Я выбираю следующую статью по списку
2. Переписываю её по ТЗ
3. Отмечаю в progress.md как [⏳] (на проверке)
4. Показываю тебе результат
5. Ты проверяешь → говоришь "ok" или даёшь правки
6. Если ok → я отмечаю [✓] и перехожу к следующей

Если прерывание:
- Я запоминаю последнюю статью [⏳]
- При продолжении смотрю progress.md
- Продолжаю с незаконченной статьи
```

### Что делать при прерывании

1. Открой файл `progress.md`
2. Найди последнюю статью с отметкой `[⏳]`
3. Скажи мне: "Продолжи с этой статьи"
4. Я возобновлю работу с этого места

### Пример файла progress.md

```markdown
# Прогресс переписывания статей

## Статьи по номеру
- [✓] 52_oludeniz_gid.md — Олюдениз (одобрено)
- [⏳] 53_babadag_gora.md — Бабадаг (на проверке)
- [ ] 54_calis_beach.md — Чалыш (не начато)
...
```

### Варианты работы

**Вариант 1: По одной с проверкой**
- Я пишу 1 статью → ты проверяешь → правки → ок → следующая
- Плюсы: контроль качества, быстрая коррекция
- Минусы: дольше по времени

**Вариант 2: Пачками по 3-5 статей**
- Я пишу несколько статий сразу → ты проверяешь все сразу
- Плюсы: быстрее
- Минусы: больше правок одновременно

**Вариант 3: Мой выбор**
- Я пишу пока не столкнусь с вопросом
- Ты вмешиваешься только при необходимости
- Плюсы: максимально быстро
- Минусы: меньше контроля

Рекомендую начать с **Варианта 1** для первых 2-3 статей настроиться, потом можно переключиться на Вариант 2.
